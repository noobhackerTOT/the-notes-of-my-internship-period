# 1

1\. Client Hello

客户端发送：支持的 TLS 版本、客户端随机数（Client
Random）、加密套件列表、扩展信息（如服务器域名 SNI）。

2\. Server Hello

服务器回复：选定的 TLS 版本、服务器随机数（Server
Random）、选定的加密套件。

3\. 服务器证书与密钥交换

服务器发送数字证书（内含公钥）。

若使用 ECDHE 等算法，还会发送 Server Key Exchange（包含临时公钥参数）。

最后发送 Server Hello Done。

4\. 客户端验证证书

客户端验证证书的合法性（有效期、域名、CA 签名、吊销状态）。

5\. 客户端密钥交换

客户端生成临时密钥对（如 ECDHE 的椭圆曲线公私钥），发送 Client Key
Exchange（包含自己的临时公钥）。

客户端与服务器各自利用对方的临时公钥和自己的临时私钥，计算出相同的
Pre‑Master Secret。

6\. 生成主密钥（Master Secret）

双方使用 PRF(Pre‑Master Secret + Client Random + Server Random) 生成 48
字节的 Master Secret。

7\. 生成会话密钥

从 Master Secret
派生出实际加密使用的密钥：客户端加密密钥、服务器加密密钥、MAC 密钥或 IV
等。

8\. 切换加密并完成握手

客户端发送 Change Cipher Spec（通知后续消息将加密），再发送 Finished
消息（加密的握手哈希值）。

服务器同样发送 Change Cipher Spec 和 Finished。

双方验证 Finished 消息无误，握手完成。

9\. 加密数据传输

应用数据（如 HTTP 请求/响应）被分割、压缩（可选）、计算 MAC、加密（如
AES‑GCM），加上 TLS 记录头后发送。

接收方解密、验证、重组，交给上层应用。

PKI 基本概念和原理

什么是 PKI？

PKI（Public Key Infrastructure，公钥基础设施）
是一个通过数字证书将公钥与实体身份进行绑定的体系，它为网络通信提供身份认证、数据加密、数字签名等安全服务。

简单理解：PKI
是一个信任中介系统，解决"我拿到的公钥到底是不是对方的"这个问题。

\(1\) 密钥管理原理

什么是密钥管理？

密钥管理是指对密钥从产生到销毁的整个生命周期进行安全、有效的管理。它是密码系统安全的基础，因为即使算法再强，如果密钥泄露或管理不当，整个安全体系都会崩溃。

密钥的生命周期

完整的密钥管理覆盖以下阶段：

阶段 主要任务 安全要求

生成 使用强随机数生成器产生密钥 随机性、不可预测性、足够的长度

分发 将密钥安全地传递给通信双方 机密性、完整性、身份认证

存储 密钥的保存（如硬件安全模块HSM、密钥库） 防泄露、防篡改、访问控制

使用 加解密、签名、验证等操作 不暴露明文密钥，限制使用次数/场景

更新 定期更换密钥（避免长期使用同一密钥） 平滑过渡，保证旧密钥安全销毁

备份与恢复 防止密钥丢失导致数据不可恢复 安全备份，多因素控制恢复

销毁 彻底删除密钥及其所有副本 不可恢复，防止事后泄露

核心安全原则

最小权限：每个密钥只用于其设计目的（如加密密钥不用于签名）。

密钥隔离：不同用途、不同系统的密钥分开管理。

密钥分离：主密钥（加密其他密钥）与工作密钥（实际加解密数据）分开。

密钥轮换：定期更换密钥，降低长期泄露风险。

可审计性：所有密钥操作应有日志，便于追踪异常。

\(2\) 密钥分配方案

密钥分配是密钥管理中最具挑战性的环节，因为密钥需要在不可信的信道上安全传输。主要有以下几类方案：

一、对称密钥分配

通信双方使用相同的密钥进行加解密，因此必须安全地共享该密钥。

1\. 预共享密钥（PSK，Pre-Shared Key）

方式：通过人工或物理方式（如U盘、面对面）提前把密钥注入双方设备。

优点：简单、无额外通信开销。

缺点：扩展性差，每对通信方需要独立密钥；密钥数量随用户数平方增长（n(n-1)/2）。

适用场景：小型网络、物联网设备、VPN预共享密钥认证。

2\. 密钥分发中心（KDC，Key Distribution Center）

原理：引入可信第三方KDC，每个用户与KDC共享一个长期密钥。当A要与B通信时，KDC为A和B生成一个临时会话密钥，并分别用各自与KDC的长期密钥加密后发给A和B。

典型协议：Kerberos（广泛应用于Windows域环境）。

进行安全通信。

优点：只需维护与KDC的少量长期密钥，扩展性好。

缺点：KDC是单点故障和性能瓶颈；所有通信密钥都经过KDC。

3\. 基于非对称加密的对称密钥分配

原理：利用公钥加密技术安全地传输对称密钥。例如：

A生成一个临时会话密钥，用B的公钥加密后发送给B，B用自己的私钥解密得到该密钥。

或者通过Diffie-Hellman密钥交换协议协商出共享密钥（无需传输密钥本身）。

优点：无需预共享密钥，适合开放网络。

缺点：依赖PKI保证公钥的真实性；公钥运算较慢。

二、非对称密钥分配

在公钥密码体系中，公钥可以公开分发，无需保密，但必须保证其真实性和完整性（即确保公钥确实属于声称的实体）。主要方案：

1\. 公开发布（Public Announcement）

方式：用户将自己的公钥通过邮件、网站等公开渠道发布。

缺点：任何人都可以伪造假公钥，无法保证真实性。

2\. 公钥目录（Public Directory）

方式：由一个可信机构维护一个公开的公钥目录，用户注册自己的公钥，其他用户可查询。

优点：比随意发布更可靠。

缺点：目录可能被篡改或假冒；仍然是单点风险。

3\. 公钥基础设施（PKI）

原理：通过数字证书将公钥与实体身份绑定，由认证机构（CA）
签发证书并签名。用户获取对方的证书后，验证CA签名即可信任该公钥。

优点：安全性高，可扩展，支持证书吊销。

缺点：需要建设和维护CA体系，成本较高。

应用：HTTPS、电子邮件加密（S/MIME）、代码签名等。

4\. 信任网络（Web of Trust）

原理：由用户相互签名对方的公钥，通过熟人关系建立信任链（如PGP）。

优点：去中心化，适合小团体或个人。

缺点：信任传递复杂，难以大规模应用。

三、混合密钥分配

实际系统中常采用混合模式：使用非对称加密（或密钥协商）安全地传输对称密钥，然后用对称密钥加密实际数据。这也是TLS/SSL、SSH、IPsec等协议的标准做法。

例如，TLS 1.2 中：

使用证书进行身份认证（非对称）

通过 ECDHE 或 RSA 协商 Pre-Master Secret（非对称/密钥协商）

生成 Master Secret 和会话密钥（对称）

后续数据传输使用 AES 等对称算法加密。
