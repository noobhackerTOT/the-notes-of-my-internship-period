Linux 查询操作命令

ls 查询目录下的文件 pwd 显示当前目录 find查找指定目录下的文件

\*通配符代表所有字符

chown user file 将文件所有者改为 user

chown user:group file 同时修改所有者和所属组

chown :group file 仅修改所属组（等同于 chgrp）

chown -R user:group dir/ 递归修改目录及其内容

find 是 Linux
中强大的文件查找命令，可在指定目录树中按条件搜索文件并执行操作。

基本语法

bash

选项 说明 示例

-name \"模式\" 按文件名查找（区分大小写） find /etc -name \"\*.conf\"

-iname \"模式\" 不区分大小写 find . -iname \"readme\*\"

-type 类型 按类型：f（文件）、d（目录）、l（链接）等 find /var -type d
-name \"cache\"

-size \[+-\]大小 按大小：c（字节）、k（KB）、M（MB） find /tmp -size
+100M

-mtime \[+-\]n 按修改时间（天），-n表示n天内，+n表示n天前 find . -mtime
-7（7天内修改）

-user 用户名 按属主 find /home -user alice

-perm 模式 按权限（如 644、-4000 等） find / -perm -4000（查找SUID文件）

-exec 命令 {} \\; 对每个找到的文件执行命令 find . -name \"\*.tmp\" -exec
rm -f {} \\;

-delete 删除找到的文件 find /tmp -name \"\*.log\" -delete

-maxdepth n 限制搜索深度 find / -maxdepth 3 -name \"passwd\"

常用示例

查找当前目录下所有 .log 文件

bash

find . -name \"\*.log\"

查找 /var 下大于 50MB 的文件

bash

find /var -type f -size +50M

查找 7 天内修改过的文件并列出详细信息

bash

find . -mtime -7 -ls

查找空目录并删除

bash

find . -type d -empty -delete

查找所有 .txt 文件并统计行数

bash

find . -name \"\*.txt\" -exec wc -l {} \\;

安全删除（先确认）

bash

find . -name \"\*.bak\" -ok rm {} \\;

技巧

使用 -print0 和 xargs -0 处理文件名含空格的文件：

bash

find . -name \"\*.jpg\" -print0 \| xargs -0 ls -l

组合多个条件：-a（与，默认）、-o（或）、!（非）

bash

find . -type f \\( -name \"\*.c\" -o -name \"\*.h\" \\) -size +1M

find 在目录树中查找文件（按文件名、类型、大小、时间、权限等条件）
文件本身（元数据）

grep 在文件内容中搜索文本模式（支持正则表达式） 文件内容

2\. uptime && free -h && df -h

命令拆分：

uptime：显示系统运行时间、登录用户数和负载。

free -h：以人类可读单位显示内存和 swap 使用情况。

Swap 的作用

缓解内存压力

当物理内存耗尽时，系统可以将部分内存页换出到 swap，避免因内存不足导致
OOM（Out Of Memory）被杀死进程。

支持内存回收与休眠

内核会主动将长期未使用的内存页（如缓存）移到
swap，以释放物理内存给文件缓存或活跃进程。

系统休眠（suspend-to-disk）时需要将内存内容保存到 swap
分区，以便断电后恢复。

应对突发内存需求

某些应用可能临时申请大量内存，交换区可以吸收这种突发请求，避免系统立即崩溃。

df -h：以人类可读单位显示文件系统磁盘空间使用率。

&& 表示前一条命令成功后才执行下一条。

输出解读：

uptime 直接给出负载，与 top 头部相同。

free -h 显示物理内存总量、已用、空闲、共享、缓冲/缓存和可用内存。

df -h 显示每个挂载点的总容量、已用、可用、使用百分比。

![](D:\docxnotes\ugreen\media/media/image2.png){width="5.583333333333333in"
height="2.3472222222222223in"}

判断：

负载过高同前。

内存：available 接近 0 且 swap used 持续增长 → 内存不足。

磁盘：使用率超过 90% 需清理，尤其 / 或 /var 分区。

三、内存与 Swap

6\. free -h && echo \"\-\--\" && vmstat 1 5

命令拆分：

free -h 同前。

echo \"\-\--\" 只是分隔线。

vmstat 1 5：虚拟内存统计，每秒输出一次，共 5 次。

vmstat 输出关键字段：

procs：r 运行队列长度，b 阻塞进程数。

memory：swpd 已用 swap，free 空闲物理内存，buff 缓冲，cache 缓存。

swap：si 从磁盘交换进内存，so 从内存交换出磁盘。

io：bi 块读，bo 块写。

system：in 中断，cs 上下文切换。

![](D:\docxnotes\ugreen\media/media/image4.png){width="5.763888888888889in"
height="1.8881944444444445in"}

判断：

若 si/so 持续非零，说明内存不足，系统频繁使用 swap。

r 长期大于 CPU 核心数表示 CPU 过载；b 大于 0 表示有进程阻塞 I/O。

cs 很高可能表示锁竞争或大量线程切换。

简单来说：

buffer 更偏向于"准备写入磁盘"的临时存储。

cache 更偏向于"已从磁盘读取"的页面缓存。

1\. si / so

si：swap in -- 每秒从交换空间（swap）换入内存的页面数（单位通常为 KB
或页）。

so：swap out -- 每秒从内存换出到交换空间的页面数。

意义：如果 si 和 so 持续非零，说明物理内存不足，系统被迫频繁使用 swap
进行页面交换，这会严重降低性能（因为磁盘 I/O 远慢于内存）。

2\. r

r：running / runnable processes -- 运行队列中的进程数。包括正在 CPU
上运行的进程以及等待 CPU 的进程。

意义：如果 r 长期大于 CPU 核心数（例如 8 核机器上 r 长期 \> 8），说明
CPU 过载，大量进程在等待 CPU 时间，导致响应延迟。

3\. b

b：blocked processes --
处于不可中断睡眠状态的进程数。这类进程通常正在等待 I/O
操作完成（如磁盘读写、网络等）。

意义：b 大于 0 表示存在进程因 I/O 阻塞，可能磁盘或网络成为瓶颈。

4\. cs

cs：context switches -- 每秒上下文切换的次数。上下文切换指 CPU
从一个进程/线程切换到另一个。

意义：cs 值很高（需结合 CPU
使用率判断）通常意味着系统存在频繁的线程切换，可能由锁竞争、过多的线程数量或不合理的内核调度引起，会消耗
CPU 开销，降低效率。

总结：这些指标是快速定位系统性能瓶颈的关键线索。

高 si/so → 内存不足；

高 r → CPU 不足；

高 b → I/O 瓶颈；

高 cs → 可能的锁争用或过度线程化。

VSZ（Virtual Memory Size）

进程的虚拟内存大小（单位通常为
KB）。包括进程可访问的所有内存，如代码段、数据段、共享库、映射的文件等。这个值可能远大于实际使用的物理内存，因为它只是地址空间的分配。

VSZ
很高不一定表示占用物理内存多，但可能暗示程序申请了大量虚拟地址空间（如数据库的共享内存、JVM
的堆预留）。

如果 VSZ 异常增大且不释放，可能是内存泄漏的前兆（虚拟内存持续增长）。

RSS（Resident Set Size）

进程的常驻内存大小（单位
KB）。表示该进程当前实际占用的物理内存（不包含被交换出去的部分）。RSS
是衡量进程真实内存占用的常用指标。

排查内存瓶颈时，RSS 是最直接的指标。

多个进程 RSS 之和可能超过物理内存，因为有共享内存重复计算。

监控 RSS 长期居高不下且持续增长，通常是内存泄漏的迹象

TTY

与进程关联的控制终端。例如 pts/0、tty1、? 等。

pts/\* 表示伪终端（如 SSH 或终端模拟器打开的窗口）。

tty1 表示物理终端（如按 Ctrl+Alt+F1 进入的文本控制台）。

? 表示没有控制终端，通常是守护进程或后台进程。

通过 TTY 可以快速定位进程的所属会话。例如，发现某个异常进程占用大量 CPU
且 TTY 为 pts/2，可以判断它是从哪个终端启动的，进而排查登录用户。

守护进程通常 TTY 为
?，如果某个服务进程意外关联了终端，可能是不正确的启动方式

硬链接（Hard Link）

定义

硬链接是指通过 inode 直接关联文件数据。每个文件都有一个唯一的
inode，硬链接本质上是为同一个 inode
创建多个目录项（文件名）。删除任一硬链接，其他链接依然有效，只有当所有硬链接被删除后，文件数据才会被释放。

特点

同一个 inode：所有硬链接共享同一个 inode 号，文件数据只有一份。

不能跨文件系统：因为 inode
在不同文件系统中是独立的，硬链接只能在同一个文件系统内创建。

不能链接目录：出于防止循环引用的安全考虑，普通用户不能创建目录的硬链接（系统硬链接如
. 和 .. 除外）。

index node（索引节点）

链接是一个特殊的文件，其内容指向另一个文件或目录的路径（类似于 Windows
的快捷方式）。它有自己的 inode 和文件属性，访问时会重定向到目标路径。

特点

独立的 inode：软链接有自己的 inode 和文件数据（存储目标路径）。

可以跨文件系统：因为仅存储路径字符串，不依赖目标文件的 inode。

可以链接目录：软链接可以指向任何类型的文件或目录。

目标删除后失效：如果原文件被删除，软链接会成为"悬空链接"（dangling），访问时会报错。

权限无关：软链接的权限通常为 lrwxrwxrwx，实际访问时由目标文件权限决定。

删除不影响其他链接：rm 只是减少链接计数，只有当链接计数降为 0
时，空间才被释放。

可重启对应进程或清空文件（\> /path/to/file）来释放空间。

进程卡死，无法正常 kill

现象：一个 Java 进程（PID 12345）占用 CPU 100%，你执行 kill 12345
后进程依然存在，变成僵尸状态（Z）。你想查看该进程到底卡在哪个系统调用上，或者它在"死等"什么资源。

问题：用什么命令跟踪该进程当前正在执行的系统调用？

System call trace

strace 能显示进程正在执行的系统调用（如 read, write, epoll_wait
等）。如果卡在某个调用上（比如 futex 死锁），就能看到进程永远停在那行。

补充：若进程已僵尸，无法 strace，则需检查父进程或重启系统。

Ps 常见与进程的静态 htop top 对应进程动态占用的资源

网络丢包或延迟大

现象：应用调用外部 API
经常超时。你怀疑是网络路径不稳定，想看看从本机到目标 IP（例如
1.1.1.1）经过哪些路由节点，每一跳的延迟和丢包情况。

mtr（My TraceRoute）

命令示例：mtr -r -c 100 1.1.1.1（报告模式，发送 100 个包）或直接运行 mtr
1.1.1.1 进入交互界面。

解析：traceroute 只能看到路径，ping 只能看端到端丢包。mtr
结合两者，显示每一跳的丢包率和延迟，是网络排查利器。

磁盘空间明明没满，但无法写入文件

现象：你在 /app/logs 目录下尝试 touch test.txt，报错 No space left on
device。但执行 df -h 显示 /app 分区还有 30% 空闲。

inode 耗尽

原因：磁盘分区的
inode（索引节点）数量用尽，即使还有空间也无法创建新文件。常见于大量小文件场景（如邮件队列、会话缓存、日志轮转失败等）。

检查命令：

df -i 查看 inode 使用情况（IUse% 列）。

若 /app 分区 inode 使用率 100%，则确认问题。

进一步定位：

find /app -type f \| wc -l 统计总文件数。

for i in /app/\*; do echo \$i; find \$i -type f \| wc -l; done
找出哪个子目录文件最多。

解决：删除大量小文件，或调整应用程序减少文件生成。

1\. for i in /app/\*; do \... done

for 循环，变量 i 依次取 /app/\* 通配符展开后的每个值。

/app/\* 会匹配 /app
目录下的所有直接子项（包括文件和子目录，不递归）。例如
/app/logs、/app/config、/app/file.txt。

2\. echo \$i

打印当前项的名称（完整路径，如
/app/logs）。相当于告诉用户下面统计的是哪个目录或文件。

3\. find \$i -type f \| wc -l

find \$i -type f：在路径 \$i 下查找所有普通文件（-type
f），递归地包括子目录中的文件。

\|：管道，将前面的输出（每行一个文件路径）传递给下一个命令。

wc -l：统计行数，即文件的数量。

如果 \$i 本身是一个普通文件（而不是目录），find
会找到该文件（因为该文件也符合 -type f），所以输出为 1。

服务端口正常监听，但外部访问超时

现象：ss -tlnp 看到 Nginx 监听 0.0.0.0:80，curl http://localhost
正常返回，但从另一台机器 telnet your-ip 80 却超时。

本地回环（127.0.0.1）能访问，但外部 IP
无法访问，通常是防火墙阻止了入站流量。

检查命令：

iptables -L -n -v 查看是否有规则 DROP 或 REJECT 目标端口的入站包。

firewall-cmd \--list-all。

云环境需检查安全组或网络 ACL（在云控制台查看）。

也可从外部执行 nmap -p 80 你的公网IP 看端口是否被过滤

大量 TIME_WAIT 连接导致端口耗尽

现象：压测后，业务出现 Cannot assign requested address 错误。ss -tan
看到大量 TIME_WAIT 状态的连接。

统计 TIME_WAIT 及内核调优

统计命令：

bash

ss -tan \| grep TIME-WAIT \| wc -l

\# 或

netstat -tan \| grep TIME_WAIT \| wc -l

内核参数调整（/etc/sysctl.conf）：

bash

net.ipv4.tcp_tw_reuse = 1 \# 允许客户端重用 TIME_WAIT 连接（安全）

net.ipv4.tcp_tw_recycle = 0 \# 已废弃且不建议开启（NAT 环境导致问题）

net.ipv4.tcp_fin_timeout = 30 \# 缩短 FIN-WAIT-2 超时（默认60）

net.ipv4.ip_local_port_range = 1024 65000 \# 扩大客户端端口范围

日志中出现 \"Out of memory: Kill process\"

现象：dmesg -T 显示 OOM Killer
杀死了某个进程。你想知道当时系统内存分配情况，以及哪个进程被杀了。

问题：用什么命令查看 OOM 发生时内核记录的内存信息？

dmesg -T \| grep -i \"out of memory\" -A 30 \# 显示 OOM 发生及后续30行

\# 或

grep -i \"killed process\" /var/log/messages

\# systemd 系统

journalctl -k \| grep -i oom

关键信息：OOM Killer 会输出：

被 kill 的进程 PID、名称、内存占用（rss、total-vm）

系统内存总量、空闲内存、各进程的分数

九、一键诊断脚本

bash

echo \"=== LOAD ===\"; uptime; \\

echo \"=== MEMORY ===\"; free -h; \\

echo \"=== DISK ===\"; df -h; \\

echo \"=== TOP CPU PROCESSES ===\"; ps aux \--sort=-%cpu \| head -5; \\

echo \"=== TOP MEM PROCESSES ===\"; ps aux \--sort=-%mem \| head -5; \\

echo \"=== RECENT KERNEL ERRORS ===\"; dmesg -T \| grep -i
\"error\\\|fail\" \| tail -10; \\

echo \"=== TCP CONNECTIONS ===\"; ss -t state established \| wc -l

说明：这是一个串行执行的命令组，使用 ; 依次输出各模块信息。

输出内容：依次显示负载、内存、磁盘、前 5 个高 CPU 进程、前 5
个高内存进程、最近 10 条内核错误、当前 TCP 连接总数。

用途：快速获取系统核心指标，适合第一轮排查。可根据需要修改 head -5
的数量或增加其他检查项。

整体资源（load、内存、磁盘）快速定位瓶颈域。

深入进程（CPU、内存、I/O）找出具体异常进程。

网络状态（连接数、流量）判断是否遭受攻击或配置问题。

日志分析（系统日志、内核日志）确认底层错误。

硬件健康（PCI、SMART、DMI）排除硬件故障。

Lsblk

  -----------------------------------------------------------------------------------------
  场景                      你通过 lsblk 能看出什么
  ------------------------- ---------------------------------------------------------------
  **查看磁盘数量**          系统里插了几块物理硬盘、几个U盘、SD卡

  **确认分区布局**          例如 mmcblk0 上有 p1(boot)、p2(rootfs)、p7(overlay)

  **识别 RAID/LVM**         绿联NAS的存储池会显示为 lvm 类型，挂载点如 /volume1

  **定位 USB 设备**         能看到 /dev/sde 及其分区 sde1、sde2，对应你之前看到的已满分区

  **检查未挂载分区**        某些分区没有 MOUNTPOINT，可能是未使用或需要手动挂载
  -----------------------------------------------------------------------------------------
