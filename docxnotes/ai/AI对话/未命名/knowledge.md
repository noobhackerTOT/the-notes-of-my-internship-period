## 一、SSH 蜜罐（asyncssh）

### 1. 为什么不用 paramiko 而是 asyncssh？paramiko + asyncio.to_thread 有什么问题？

**解答**：

- ​`paramiko` 是同步阻塞库：每个 SSH 连接会占用一个系统线程（或 Python 线程），当需要同时处理数百个连接时，线程数量膨胀 → 内存开销大（每个线程约 8MB 栈内存）、上下文切换频繁、GIL 争用严重。
- ​`asyncio.to_thread`​ 虽然可以将阻塞操作转移到线程池，但仍然受限于线程池大小（通常 CPU 核心数 × 5），且每个请求都要跨越事件循环和线程之间的边界，带来序列化/同步开销。更重要的是，`paramiko` 内部有自己的 socket 循环，丢进线程池后，无法与 asyncio 的其他组件（如 WebSocket、HTTP 蜜罐）共享同一个事件循环，导致整体架构割裂。
- ​`asyncssh`​ 是原生 asyncio 实现：整个协议栈（TCP、握手、认证、通道）都用 `async/await` 编写，可以和其他 asyncio 服务无缝合并在同一个线程的事件循环中运行，单进程轻松支撑数千并发连接，且代码风格一致。

### 2. `validate_password`​ 返回 False，但异步上报。如果 `_report` 异常，握手会失败吗？

**解答**：

- ​`asyncio.create_task`​ 会启动一个独立的协程任务，默认不会等待它的结果。如果 `_report`​ 内部抛出异常，该异常不会传播到 `validate_password` 中，因此不会中断 SSH 握手。
- 但未捕获的异常会被事件循环捕获并记录到日志（可能触发 `Task exception was never retrieved`​ 警告）。正确做法是在 `_report`​ 内部用 `try/except` 包裹所有上报逻辑，至少记录错误日志，确保不上报行为不影响蜜罐的可用性。

### 3. Docker 中 `%TEMP%/threathive_ssh_host_key` 指向哪里？如何保证重启不丢失？

**解答**：

- ​`%TEMP%`​ 是 Windows 环境变量，在 Linux 容器中无意义。代码中应该用 `tempfile.gettempdir()`​ 或直接使用 `./data/ssh_host_key` 这样的相对路径。如果使用临时目录，容器重启后数据会丢失。
- 为了保证密钥持久化，应通过 Docker 卷（volume）将宿主机目录挂载到容器内的固定路径，例如：`-v ./data:/app/data`​，并在代码中写入 `/app/data/ssh_host_key`。

### 4. `asyncssh.create_server` 如何利用事件循环避免阻塞？

**解答**：

- ​`asyncssh`​ 内部使用 asyncio 的 `start_server`​ 创建 TCP 服务，并为每个连接创建一个 `SSHConnection` 协程对象。所有套接字 I/O 都注册到事件循环的文件描述符监视器中，当数据到达时事件循环回调对应的协程继续执行。
- SSH 协议的状态机（解密、加密、消息分帧）都是非阻塞的，每个协程在 `await` 读写数据时主动让出 CPU，事件循环可以处理其他连接。因此，数百个连接同时存在时，只有一个线程交替执行各个协程，没有线程上下文切换开销。

## 二、HTTP 蜜罐（aiohttp）

### 5. `await request.read()` 遇到 1GB POST body 会怎样？有限制吗？

**解答**：

- 默认 `aiohttp` 服务器会读取整个请求体到内存，1GB 数据将导致内存暴涨，可能 OOM 杀死容器。
- 应在中间件或路由处理前设置限制：`web.Application(client_max_size=10_485_760)`​ 限制 10MB，或使用 `request.content.read(limit=1024*1024)` 只读前 1MB。对于蜜罐，完全没必要读取超大 body，超限后直接记录元数据即可。

### 6. `/.env` 的假凭据是静态还是动态？被复用会造成误报吗？

**解答**：

- 在示例实现中通常是**静态**假凭据（如 `AWS_ACCESS_KEY=AKIAFAKE...`），目的是诱使攻击者复制使用。如果攻击者真的拿去尝试登录，可能会对第三方系统产生无效登录日志，但这属于攻击者的行为，不是蜜罐的误报。真正的误报是蜜罐将自己的假凭据识别为真实 IOC 并告警——只要不把假凭据加入 IOC 列表即可。

### 7. 不同端点返回不同 `Server` 头，攻击者能识别蜜罐吗？

**解答**：

- 可以。真实的 nginx 或 Apache 服务器会在所有响应中返回一致的 `Server`​ 头，而蜜罐根据路径动态变化，这本身就是一种指纹特征。更隐蔽的做法是统一返回同一个 `Server`​ 头（如 `nginx/1.18.0`），或者干脆不返回（但有些扫描器会探测缺失行为）。

## 三、MySQL 蜜罐（手写线协议）

### 8. 20 字节盐值如何生成？随机还是固定？

**解答**：

- 标准 MySQL 握手要求两个 8 字节盐（plugin data part1+part2），通常随机生成并缓存在会话中。随机生成能防止攻击者预计算彩虹表。
- 随机不会影响客户端计算，因为客户端会读取握手包中的盐值并直接使用。蜜罐需要保存本次连接的盐值，以便后续验证（虽然蜜罐不验证）。但为了简单，许多蜜罐使用固定盐值，这降低安全性但影响不大，因为蜜罐并不真正认证。

### 9. 能否抓到明文密码？

**解答**：

- 不能。MySQL 原生认证使用 `SHA1( password ) ^ SHA1( salt + SHA1( SHA1( password ) ) )` 传输，无法逆向得到明文。但蜜罐可以记录哈希，离线跑字典。弱密码（如 123456）的哈希值可被预先计算（彩虹表）识别。
- 若要捕获明文，可以强制要求客户端使用 `mysql_clear_password` 插件（不安全），但那样攻击者可能断开连接。

### 10. `SHOW VARIABLES` 会怎样响应？

**解答**：

- 蜜罐应返回 `ERR(1045)`​ 拒绝访问，这会导致客户端认为认证失败而断开。更好的做法是返回一个伪造的结果集（如 `version_comment = "MySQL 5.7.33"`），让攻击者以为连接成功，从而继续发送更多查询。但实现 result set 编码较复杂，可以在后续迭代中加入。

## 四、Redis 蜜罐（RESP）

### 11. 解析时长度不匹配会崩溃吗？

**解答**：

- 健壮的实现不会崩溃。解析 Multi-bulk 时，提取 `$len`​ 后检查实际数据字节数是否足够。如果不足则等待更多数据（返回 `None`​ 表示不完整）；如果超出或不足但数据已接收完，则记录协议错误并关闭连接（发送 `-ERR protocol error\r\n` 后断开）。

### 12. CONFIG SET 分两次发送，检测逻辑能工作吗？

**解答**：

- 如果检测逻辑放在 `CONFIG SET`​ 命令执行前，对每个命令单独检测，那么第一次 `CONFIG SET dir /tmp`​ 可能不触发告警（除非 `/tmp`​ 也是敏感路径），第二次 `CONFIG SET dbfilename authorized_keys`​ 会触发 `ssh_key_injection`​ 告警。但如果攻击者先设置 `dir /root/.ssh`​，再设置 `dbfilename authorized_keys`，两次都会触发。建议在检测时关联同一 IP 的前后命令，但简单实现中单独检测已足够。

### 13. AUTH wrongpass 后还能继续执行命令吗？

**解答**：

- Redis 默认情况下，AUTH 失败会返回 `-ERR invalid password`​ 并**保持连接**（客户端可以重试）。蜜罐应该模拟这一行为：记录错误密码，但继续接受后续命令（包括 CONFIG SET）。除非命令要求认证（通常无认证模式），否则不应断开。

## 五、蜜罐基类（模板方法）

### 14. SSH 重写 `start()` 违背模板方法模式吗？

**解答**：

- 严格来说，是的。模板方法模式期望基类定义不可变骨架，子类只实现钩子方法（如 `handle_session`​）。但 SSH 蜜罐需要调用 `asyncssh.create_server`​ 而不是 `asyncio.start_server`​，完全改变了服务启动方式。这属于**妥协**：因为 SSH 协议复杂，使用第三方库更安全。实际工程中，可以让基类的 `start()`​ 定义更抽象，比如 `self._create_server(handler)`，子类覆盖该方法即可。

### 15. `handle_session()` 抛出异常，连接能正确关闭吗？

**解答**：

- 在 `_handle_client`​ 中，应该用 `try/finally`​ 或 `async with` 上下文管理器保证清理。例如：

  ```python
  try:
      await self.handle_session(reader, writer)
  except Exception:
      logger.exception("Unhandled error")
  finally:
      writer.close()
      await writer.wait_closed()
  ```
- ​`server.close()`​ 后调用 `wait_closed()`​ 是为了等待所有底层传输完全关闭并释放端口。如果缺少 `wait_closed`，重新启动服务器时可能遇到“地址已占用”错误。

## 六、实时数据管道

### 16. `aiosqlite` 每秒 1000 事件能撑住吗？有锁竞争吗？

**解答**：

- ​`aiosqlite` 底层是同步 sqlite3 + 线程池 + asyncio 队列，每个写操作都会串行化。每秒 1000 条事件（约 10ms 一条）通常可以承受，但遇到复杂查询时可能排队。写入锁竞争天然存在，因为 SQLite 只支持单写事务。优化方案：批量写入（每 50 条或 0.5 秒一次 flush），或换用 PostgreSQL/asyncpg。

### 17. Redis Pub/Sub 消息堆积问题

**解答**：

- 默认 `client-output-buffer-limit pubsub 32mb 8mb 60`​ 表示如果订阅者输出缓冲区超过 32MB 或持续超过 8MB 达 60 秒，Redis 会强制断开该客户端。如果 WebSocket 广播慢，消息会在 Redis 中堆积。ThreatHive 应该配置 `config set client-output-buffer-limit 'pubsub 256mb 128mb 120'` 提高阈值，同时确保广播协程处理及时。

### 18. WebSocket 断开导致广播任务退出？

**解答**：

- 正确的写法：

  ```python
  for ws in ws_clients:
      try:
          await ws.send_json(event)
      except WebSocketDisconnect:
          ws_clients.discard(ws)
  ```

  异常只影响单个客户端，循环继续。如果异常没有被捕获，整个 `broadcast_events` 任务会崩溃。必须用 try/except 包裹发送代码。

## 七、行为分析引擎

### 19. `{{7*7}}` 会被执行吗？

**解答**：

- 不会。只做正则匹配，不渲染。真正的 SSTI 检测需要实际将 payload 注入模板并观察输出（如 `7*7=49`），但这有安全风险（可能执行任意代码）。蜜罐中更安全的方式是匹配已知 SSTI 特征，并给予中等置信度（0.7）。

### 20. 扫描检测阈值 1 分钟 10+ 事件，漏掉分布式扫描？

**解答**：

- 会漏掉。单个低频扫描（100 个 IP 每个发 1 个请求）不会触发单 IP 阈值，但可以通过**全局请求速率异常**或**频繁变化的小众 User-Agent** 来辅助检测。不过对于蜜罐而言，分布式扫描每个 IP 单独分析时通常不被标记为扫描器，而是普通探测。阈值基于经验：人工扫描器通常快速发出多个请求。

### 21. SSH 蜜罐中 `action` 字段如何生成？如何区分登录和执行命令？

**解答**：

- ​`validate_password`​ 被调用时，生成 `action="login_attempt"`​。`exec_requested`​ 被调用时，生成 `action="cmd:ls"`。
- 因为蜜罐从不接受认证，所以永远不会出现“登录成功”事件，所有命令执行请求也会被拒绝。但攻击者可能在同一个 TCP 连接中不断尝试不同密码，每次都会产生 `login_attempt`​ 事件。文档中“不成功认证”就是通过始终返回 `False` 实现的。

## 八、IOC 自动提取

### 22. IPv4 正则会匹配 `999.999.999.999` 吗？

**解答**：

- 不会。正则中 `(25[0-5]|2[0-4]\d|[01]?\d\d?)`​ 确保每一段最大 255（25[0-5] 覆盖 250-255，2[0-4]\d 覆盖 200-249，[01]?\d\d? 覆盖 0-199）。`999` 无法被任何分支匹配。

### 23. 私有 IP 为什么排除？内网横向移动怎么办？

**解答**：

- 默认排除是为了减少误报（公网蜜罐几乎不会收到私有 IP 的请求，因为源 IP 经过 NAT 后变成公网）。但如果蜜罐部署在企业内网，私有 IP 正是横向移动的重要线索。因此应该做成配置项：`EXCLUDE_PRIVATE_IPS=false` 允许采集。

### 24. MD5/SHA1 正则误提取普通十六进制串如何过滤？

**解答**：

- 长度过滤是第一步：MD5 要求正好 32 位十六进制，SHA1 40 位。但像 `deadbeef`​ 只有 8 位不会被匹配。常见的误报如 32 位序列号也会被提取，因此需要额外黑名单（排除常见静态内容如 `00000000000000000000000000000000`）或上下文检查（例如出现在 password 字段中才认为是哈希）。

## 九、威胁情报与告警

### 25. AbuseIPDB 缓存 24 小时，缓存期内被标记恶意不更新，如何平衡？

**解答**：

- 缓存会延迟情报更新。更好的方法是使用**短缓存**（1 小时）配合**主动失效**：当本地攻击频率突然增高时，强制刷新缓存。或者采用“写穿”策略：每次查询前检查缓存是否过期，过期则异步更新但先返回旧值。对于蜜罐，24 小时延迟通常可接受，因为攻击者 IP 不会瞬间从干净变恶意。

### 26. 事件次数 10 次就给满 20 分，合理吗？

**解答**：

- 有争议。10 次尝试可能是自动化扫描而不是严重攻击。但评分公式可以根据实际需求调整权重，例如将系数改为 `min(event_count * 1, 15)`​，或采用 log 函数 `min(5 * log10(event_count+1), 20)` 让初期增加快，后期平缓。

### 27. 多协议攻击告警窗口滑动还是固定？

**解答**：

- 应该是**滑动窗口**。实现方式：为每个 IP 维护一个按时间排序的协议事件列表，每次新事件插入时删除超过 3600 秒的老事件，然后检查 `len(set(protocols_in_window)) >= 3` 触发告警。固定窗口会漏掉跨边界的攻击。

## 十、部署与运维

### 28. 为什么蜜罐不依赖 API 容器就绪？

**解答**：

- API 容器负责提供 REST 和 WebSocket 服务，不影响蜜罐捕获事件。蜜罐可以通过 Redis 缓存上报事件，即使 API 未启动，Redis 也会暂存消息（Pub/Sub 会丢失，但可用 List 或 Streams 做持久化）。更好的设计是蜜罐依赖 Redis 即可，API 独立启动。所以依赖链只包含 Redis 是合理的。

### 29. `pip install -e .` 缺少依赖怎么办？

**解答**：

- ​`pyproject.toml`​ 中 `dependencies`​ 字段必须声明所有运行时依赖（如 `asyncssh>=2.13`​、`aiohttp>=3.8`​ 等）。执行 `pip install -e .`​ 时会自动安装这些依赖。如果用户只克隆代码而没有安装，手动运行 Python 文件会报 `ModuleNotFoundError`。

### 30. `demo_data.py` 会写入真实蜜罐吗？如何不影响统计？

**解答**：

- 它应该直接调用 `collector.add_event()`​ 或操作数据库，写入 `attacks`​ 表。如果不希望影响统计，可以在生成演示数据时设置一个特殊标记（如 `is_demo=True`​），并在查询统计时过滤掉 `is_demo=1` 的事件。或者，在独立的演示数据库文件中生成数据，不干扰生产库。
