# 文件系统重建

这是一个非常好的学习方向。很多人只会复制命令，但真正做恢复时，​**参数含义决定了你是在救数据还是在毁数据**。

下面我按实战最常用的工具解释参数。

---

# 一、e2fsck 参数详解（Ext4）

```
e2 + fs + ck
```

其中：

- **e2** \= ​**Ext2**（Second Extended Filesystem，第二代扩展文件系统）
- **fs** \= ​**File System**（文件系统）
- **ck** \= ​**Check**（检查）

基本语法：

```bash
e2fsck [参数] 设备
```

例如：

```bash
sudo e2fsck -f -y /dev/sdb1
```

---

## -f （Force）

强制检查。

正常情况下：

```text
文件系统是干净的
```

e2fsck会直接退出。

加：

```bash
-f
```

后：

```text
即使文件系统标记为Clean
也重新扫描所有inode和block
```

适用于：

- 怀疑磁盘异常
- 怀疑元数据损坏

---

## -n （No）

只检查，不修改。

```bash
sudo e2fsck -n /dev/sdb1
```

含义：

```text
发现问题
↓
询问修复
↓
全部回答No
```

取证分析常用。

---

## -y （Yes）

自动修复。

```bash
sudo e2fsck -y /dev/sdb1
```

相当于：

```text
发现问题？
Yes

发现下一个？
Yes

继续？
Yes
```

风险：

可能导致：

```text
lost+found增加
目录被删除
部分文件丢失
```

---

## -p （Preen）

自动修复简单错误。

```bash
sudo e2fsck -p /dev/sdb1
```

只修复：

- bitmap错误
- inode计数错误

不会修复杂项。

---

## -c （Check bad blocks）

扫描坏块。

```bash
sudo e2fsck -c /dev/sdb1
```

内部调用：

```bash
badblocks
```

把坏块加入坏块表。

---

## -cc

读写测试坏块。

```bash
sudo e2fsck -cc /dev/sdb1
```

比：

```bash
-c
```

更严格。

但速度极慢。

---

## -D

优化目录结构。

```bash
sudo e2fsck -D /dev/sdb1
```

作用：

```text
重建目录索引
优化HTree
```

适合：

```text
数百万文件
NAS
文件服务器
```

---

## -b

指定备用超级块。

例如：

```bash
sudo e2fsck -b 32768 /dev/sdb1
```

意思：

```text
不用主Superblock
改用32768号备用Superblock
```

---

# 二、Btrfs 参数详解

---

# btrfs check

检查文件系统。

```bash
sudo btrfs check /dev/sdb1
```

只检查：

```text
Root Tree
Chunk Tree
Extent Tree
Metadata
```

不修改。

---

## --repair

危险参数。

```bash
sudo btrfs check --repair /dev/sdb1
```

作用：

```text
尝试修复树结构
重建引用
删除坏记录
```

官方警告：

```text
仅在开发者建议下使用
```

---

## --readonly

只读检查。

```bash
sudo btrfs check --readonly /dev/sdb1
```

用于：

- 取证
- 生产环境

---

# btrfs rescue

---

## zero-log

```bash
sudo btrfs rescue zero-log /dev/sdb1
```

作用：

删除：

```text
Log Tree
```

解决：

```text
failed to replay log tree
```

---

## super-recover

```bash
sudo btrfs rescue super-recover /dev/sdb1
```

作用：

```text
扫描备用超级块
恢复主超级块
```

---

## chunk-recover

```bash
sudo btrfs rescue chunk-recover /dev/sdb1
```

作用：

```text
重建Chunk Tree
```

实际上：

```text
扫描整个磁盘
重新建立逻辑地址映射
```

耗时很长。

---

# mount 参数

---

## ro

只读挂载。

```bash
mount -o ro
```

等于：

```text
禁止写入
```

恢复第一步。

---

## degraded

降级挂载。

```bash
mount -o degraded
```

适用于：

```text
RAID1
RAID10
```

坏了一块盘。

---

## usebackuproot

```bash
mount -o usebackuproot
```

作用：

```text
使用旧事务中的Root Tree
```

相当于：

```text
时间回滚
```

---

## recovery

```bash
mount -o recovery
```

尝试恢复挂载。

旧版本常见。

---

# btrfs scrub

---

## start

```bash
btrfs scrub start /mnt
```

含义：

```text
扫描所有数据
验证checksum
```

---

## -B

```bash
btrfs scrub start -B /mnt
```

B：

```text
Block
```

等待执行完成。

否则后台运行。

---

## -R

```bash
btrfs scrub start -R /mnt
```

显示实时统计。

---

# ddrescue 参数详解

恢复磁盘最重要工具之一。

---

## -n

```bash
ddrescue -n
```

快速复制。

跳过坏块。

---

## -r3

```bash
ddrescue -r3
```

坏块重试：

```text
3次
```

---

## -d

```bash
ddrescue -d
```

直接访问设备。

绕过缓存。

---

## mapfile

```bash
ddrescue /dev/sdb image.dd mapfile
```

mapfile记录：

```text
哪些块成功
哪些块失败
```

下次继续恢复。

---

# 实战口诀

对于恢复来说：

```text
先镜像
后检查

先只读
后修复

先导出
后重建

Ext4：
e2fsck -n
↓
e2fsck -f
↓
e2fsck -y

Btrfs：
check
↓
zero-log
↓
super-recover
↓
chunk-recover
↓
restore
↓
--repair
```

这个顺序基本符合企业运维、NAS 数据恢复和数字取证中的常见流程，可以最大程度降低二次损坏风险。
